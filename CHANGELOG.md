### Changed
- **Scope Ring Model**: a Resource can now sit in multiple overlapping scopes simultaneously (Refs #24, closes the Core Engine "Scope Ring Model" item; **unblocks** Cross-Scope Evidence Reuse — the next #24 item). Previously every Resource had exactly one `SCOPED_TO` edge and `lemma scope discover` silently degraded multi-scope matches to "first match wins" with a yellow warning. After this PR a payments DB matching both `prod-us-east` and `pci-dss` lands as **one Resource node with two `SCOPED_TO` edges**, walked by `lemma graph impact resource:<id>` to surface the union of all reachable controls. **Breaking schema change**: `resources/*.yaml` files declaring `scope: <name>` (singular) must rename to `scopes: [<name>]` (a one-element list); single-scope resources are unchanged in behavior, just renamed. The strict-`extra="forbid"` Pydantic schema produces an "extra inputs are not permitted" error naming the offending key, so operators get a one-line fix without a hand-rolled migration warning. The redundant `scope` node attribute is dropped — edges are now the only source of truth (`get_edges` is already the consumption pattern in `scope_dot`, `scope_posture`, etc.). `ComplianceGraph.add_resource` signature changes from `scope: str` to `scopes: list[str]`; idempotent rebuild semantics preserved (re-adding with `scopes=[a, c]` after `scopes=[a, b]` cleanly drops `b`). The matcher (`scope_matcher.scopes_containing`) was already returning `list[str]` since v0; the only blocker was downstream consumers, which this PR fixes. Service at `src/lemma/services/knowledge_graph.py::add_resource` + model at `src/lemma/models/resource.py`; tests in `tests/services/test_knowledge_graph.py::TestAddResource` (8 tests including N-edge creation, multi-scope rotation idempotency, all-or-nothing scope-resolution, empty-list rejection) and `tests/models/test_resource.py` (8 tests including the old-`scope:` rejection path and `min_length=1` enforcement). **Three absorbed features ship in the same PR rather than as follow-ups**: edge attribution (every `SCOPED_TO` edge carries `matched_rules: [{source, operator, value}, ...]` recording which scope rules fired during discover; manual declarations carry an empty list); the new `lemma scope explain <resource-id>` command that renders the per-scope attribution as a readable explanation (`prod ← aws.tags.Environment, equals, 'prod'`); and a new `ComplianceGraph.outgoing_edges` primitive used by `explain` and useful elsewhere. Mechanical fan-out: every existing `add_resource(scope=)` call site (11 discover services, 10+ test files) renamed to `scopes=[...]`. After this slice, the remaining Core Engine items on #24 are Cross-Scope Evidence Reuse, HCL parser, and Continuous Validation.
- `lemma query` now reaches every edge type in the compliance graph (Refs #97). The translator prompt previously capped `edge_filter` to the Phase 2 three (`SATISFIES`, `HARMONIZED_WITH`, `CONTAINS`) — a contradiction with the schema-summary block five lines below that already advertised every edge — so every relationship shipped in #76 (`EVIDENCES`, `SCOPED_TO`, `OWNS`, `IMPACTS`, `THREATENS`, `MITIGATED_BY`, `APPLIES_TO`) was invisible at the user surface. The prompt now relies on the dynamic schema summary for the vocabulary and appends a compact cheatsheet naming one example question per edge with explicit direction hints for the ones that need them (`OWNS` from a Control uses `direction=in`, etc.) plus a disambiguation note contrasting `IMPACTS` (Resource→Control) with `MITIGATED_BY` (Risk→Control). Query result rendering gains an "Attributes" column: Evidence rows show `producer · time_iso`, Risk rows show `title [SEVERITY]`, Person shows email, Resource shows its type — auditors stop hand-joining the graph export to read a single result. Filter infrastructure (time range, severity, connector source) and a dedicated `evidence_query` trace discriminator stay tracked on #97.

### Added
- **First-party Jira connector** (Refs #115, advances #27 connector roster). Pulls change-management evidence from a Jira Cloud site and emits one `ComplianceFinding` aggregating the approved / rejected breakdown across recent change tickets — the auditable signal a SOC 2 CC8.1 review wants for change management. **Surface**: `JiraConnector(base_url, email, token, jql=DEFAULT, client=None)` at `src/lemma/sdk/connectors/jira.py`. **Default JQL**: `labels = "change-management" ORDER BY created DESC` — operators override via the `jql` config field for per-project / per-time-window scoping. **Auth**: Jira Cloud uses HTTP Basic with `email:api_token`; token via `LEMMA_JIRA_TOKEN` env var or `token=...` constructor arg. Missing email or token is a loud error at construction time, not a cryptic 401 mid-collect. Rate-limited responses (HTTP 429) raise a clean `ValueError` naming the endpoint. **Stable `metadata.uid`** per `(site, UTC date)` so same-day re-runs dedupe against themselves through the EvidenceLog's per-day dedup. **Status mapping**: `status_id=1` (Pass) when at least one approved change is present, `status_id=2` (Fail) when changes exist but none are approved, `status_id=0` (Unknown) when the JQL matched zero tickets. **CLI wiring**: registered in `_first_party_connector` factory at `src/lemma/commands/evidence.py`; works with both the legacy positional+flag form and the `lemma evidence collect --config <path>` mode from #116. **Test surface**: 9 cycles in `tests/sdk/test_jira_connector.py` (token required, email required, change-management finding shape, empty-result emits status 0, rate-limit raises, Basic auth header populated correctly, JQL override propagates, dedupe UID stable per site+date, end-to-end smoke through `lemma evidence collect --config`). - **`lemma evidence collect --config <path>` — connector-level configuration file** (Refs #116, **closes #116**). Adds the `lemma_connector_config.yaml` format that captures connector configuration (endpoints, tokens, filters) distinctly from code, so operators can reconfigure a connector without editing Python and secrets can live outside the Python module via `${ENV_VAR}` interpolation. **Schema** (`src/lemma/services/connector_config.py::ConnectorConfig`, strict Pydantic): `connector` (required, first-party name), `config` (required, dict passed to the connector factory), `producer` (optional, default empty), `enabled` (optional, default true), `schedule` (optional cron-like string for downstream schedulers). Top-level keys are `extra="forbid"` so typo'd keys (e.g. `schedules` instead of `schedule`) fail validation immediately with a clear error naming the offender. **Env-var interpolation**: `${VAR}` in any string value (top-level or nested in dicts/lists) is replaced at load time from `os.environ`. **Strict on missing**: an unset env var raises `ValueError` rather than silently substituting an empty string — silently producing an empty token field is the exact footgun an operator would never catch. **CLI**: `lemma evidence collect --config path/to/lemma_connector_config.yaml` is the new mode; the existing positional + per-connector flags (`--repo`, `--domain`, `--region`) remain backward compatible. When `--config` is set, CLI flags are ignored (the file is the source of truth so an operator can't accidentally mix file-declared and CLI-declared values). `enabled: false` skips the run cleanly with a yellow advisory. **Test surface**: 10 new service cycles in `tests/services/test_connector_config.py` (minimal YAML, full shape, env-var interpolation, unknown env var raises, nested-string interpolation, missing connector key, missing file, extra top-level keys rejected, invalid YAML, disabled flag preserved); 5 new CLI cycles in `tests/commands/test_evidence.py::TestCollectFromConfigFile` (config file drives the run, env-var interpolation works through the CLI, disabled exits 0 with advisory, missing config file exits 1, legacy positional+flag form still works). **Out of scope, deferred**: `Connector.from_config(cfg)` factory classmethod for SDK-defined connectors (current MVP routes through the existing `_first_party_connector` factory; SDK extension lands when a non-first-party connector needs it); pluggable secret backends beyond env-var interpolation (OS keyring, Vault, AWS Secrets Manager — tracked under #117); per-connector config sections in a single file (current shape is one connector per file). - **Prometheus `/metrics` endpoints on `lemma-agent serve` and `lemma control-plane serve`** (Refs #25). Closes the "Prometheus-style federation" task on #25. Both processes already exposed `/health` for human-readable status; this PR adds `/metrics` returning Prometheus exposition format (`text/plain; version=0.0.4; charset=utf-8`) so a single Prometheus scraper can pull federation-wide signals from every Lemma component without bespoke instrumentation. **Agent-side metrics** (`lemma-agent serve`): three series — `lemma_agent_uptime_seconds` (gauge), `lemma_agent_evidence_total` (counter, disk-derived from non-blank lines in `--evidence-dir/*.jsonl`), `lemma_agent_producers` (gauge, subdirectories of `--keys-dir` that contain a `meta.json`). - **`lemma control-plane serve` — Control Plane receiver** (Refs #25). Closes the central conceit of #25: an agent → Control Plane federation loop is now operable end-to-end with Lemma alone. The receiving side accepts signed envelopes POSTed by `lemma-agent forward`, verifies them against the producer's public key, and persists them to a per-producer day file. **Surface**: `lemma control-plane serve --port <N> --evidence-dir <dir> --keys-dir <dir> [--bind ADDR] [--cert FILE --key FILE] [--client-ca FILE]`. **Endpoints**: `POST /v1/evidence` (body: one signed envelope JSON, matches one line of `forward`'s output) returns `200 {"verdict":"PROVEN","entry_hash":...}` on success, `422 {"verdict":"VIOLATED","reason":...}` (or `DEGRADED`) on verification failure, `400` on malformed JSON; `GET /health` returns a JSON snapshot mirroring the agent's `/health` shape (evidence_count + producer_count + started_at + uptime_seconds) so a single observability probe works for both sides of the federation. **Per-producer chain isolation**: each producer gets its own subdirectory under `--evidence-dir`; chain verification (`prev_hash` → prior `entry_hash`) is per-producer, so cross-producer corruption is impossible by construction. **Persistence-then-verify**: envelopes are appended to disk before verification runs, leaving suspicious envelopes on disk for forensic review. **TLS + mTLS**: `--cert` + `--key` together enable HTTPS via `ssl.SSLContext.wrap_socket`; `--client-ca` adds `CERT_REQUIRED` client-cert verification (mutual TLS). Three security guard-rails: cert+key paired (passing one without the other exits 1), client-ca requires cert (mTLS only over HTTPS, exits 1 otherwise), bind defaults to `127.0.0.1` (operators must explicitly pass `--bind 0.0.0.0` to expose). **New service**: `src/lemma/services/control_plane.py::make_handler_class` returns a stdlib `BaseHTTPRequestHandler` subclass closed over evidence + keys directories — no new HTTP framework dependency, just `http.server`. The handler reuses `EvidenceLog.verify_entry` (chain + content hash + signature + lifecycle merge from #193) so receiver-side verification is byte-for-byte parity with `lemma evidence verify`. **New CLI command group**: `lemma control-plane` registered in `src/lemma/cli.py`. **Test surface**: 8 cycles in `tests/services/test_control_plane.py` (per-producer day-file persistence, malformed JSON rejection, missing-producer-key rejection, tampered-signature detection, unknown-path 404, /health endpoint, two-envelope chain consistency, cross-producer subdirectory isolation); 5 cycles in `tests/commands/test_control_plane.py` (help lists serve, missing required flags exit non-zero, cert-without-key exits 1, client-ca-without-cert exits 1, end-to-end serve + POST + 200 PROVEN). **Out of scope, deferred to later #25 slices**: per-agent rate limiting / backpressure, aggregation across ≥ 2 agents into a unified compliance graph (the AC for that), the policy-push direction (Control Plane → Agent), retry-friendly bulk ingestion, durable shutdown semantics, Prometheus-style federation. The receiver is intentionally minimal in v1 — it's the missing federation surface, not a fully-featured Control Plane. Updated `docs/reference/index.md` with a new `## lemma control-plane` section documenting the surface, endpoints, per-producer isolation, persistence semantics, and the deferred items. No new dependencies (Python: stdlib `http.server` + `ssl`). - **`lemma-agent keyrotate` and `lemma-agent keyrevoke` — full key-lifecycle parity in Go** (Refs #25 Slice N). ### Changed
- **Breaking:** `EvidenceLog`'s signed-hash algorithm now covers the pre-storage provenance records in addition to the event body. Specifically, `entry_hash = SHA-256(prev_hash || canonical_json({"event": event, "provenance": pre_storage_records}))`. Envelopes written before #99 were hashed over the event alone and will verify as VIOLATED under the new algorithm — regenerate local `.lemma/evidence/*.jsonl` files after upgrading. `EvidenceLog.append()` gains a keyword-only `provenance: list[ProvenanceRecord] | None = None` parameter; callers that don't pass it continue to get a storage-only envelope.
