## [Unreleased]

### Changed
- `lemma query` now reaches every edge type in the compliance graph (Refs #97). The translator prompt previously capped `edge_filter` to the Phase 2 three (`SATISFIES`, `HARMONIZED_WITH`, `CONTAINS`) — a contradiction with the schema-summary block five lines below that already advertised every edge — so every relationship shipped in #76 (`EVIDENCES`, `SCOPED_TO`, `OWNS`, `IMPACTS`, `THREATENS`, `MITIGATED_BY`, `APPLIES_TO`) was invisible at the user surface. The prompt now relies on the dynamic schema summary for the vocabulary and appends a compact cheatsheet naming one example question per edge with explicit direction hints for the ones that need them (`OWNS` from a Control uses `direction=in`, etc.) plus a disambiguation note contrasting `IMPACTS` (Resource→Control) with `MITIGATED_BY` (Risk→Control). Query result rendering gains an "Attributes" column: Evidence rows show `producer · time_iso`, Risk rows show `title [SEVERITY]`, Person shows email, Resource shows its type — auditors stop hand-joining the graph export to read a single result. Filter infrastructure (time range, severity, connector source) and a dedicated `evidence_query` trace discriminator stay tracked on #97.

### Added
- `lemma scope discover file --path <csv-or-json>` — **first on-prem auto-discovery source for #24**, opening the on-prem section after the Cloud section closed with #149. Reads a hand-curated or exported list of resources from a local file (CSV / JSON / JSONL) and runs each one through the existing scope matcher → graph-write loop. Air-gapped story for environments without outbound internet (DoD, on-prem hospitals, classified envs); also a universal escape hatch for any CMDB / spreadsheet / Ansible inventory that can produce a flat resource list. Format detection by extension: `.json` → top-level array, `.jsonl` → one record per line, `.csv` → header row + data rows. CSV columns expand via dotted paths — `vsphere.tags.environment` becomes `attributes["vsphere"]["tags"]["environment"]`, so operators can write CSV that produces matcher-compatible nested attributes without hand-constructing JSON. Operator owns the schema: unlike the cloud providers that auto-wrap attributes under `aws.*` / `gcp.*` / `k8s.*` / `azure.*`, file import uses the operator's `id` / `type` / `attributes` verbatim — no auto-prefixing, no auto-namespacing. Validation is fail-loud: missing `id` or `type` aborts with the offending record index; duplicate ids within a file abort with every duplicate listed; malformed JSON / JSONL aborts with file/line. Empty files return zero results, exit 0. Reuses the existing `--path` option already used by the `terraform` provider — consistent "local file source" mental model. Service at `src/lemma/services/file_discovery.py`; CLI extension on `discover_command` slots the `file` branch in alongside the cloud providers. No new dependencies (stdlib `csv` + `json`). Opens the on-prem section of #24's auto-discovery checklist; vSphere / Ansible / ServiceNow CMDB / Device42 / network scanning remain.
- `lemma scope discover azure` — **fifth and final cloud auto-discovery source for #24**, completing the entire Cloud Auto-Discovery section after AWS-API (#145), Terraform-state (#146), Kubernetes (#147), and GCP (#148). Walks Azure via **Resource Graph** (`Microsoft.ResourceGraph`) — Azure's canonical "what resources exist" API, equivalent to GCP's Cloud Asset Inventory — using the official `azure-mgmt-resourcegraph` SDK. One client + one KQL query covers Compute, Storage, IAM, beating per-service-SDK fan-out into `azure-mgmt-compute` / `azure-mgmt-storage` / `azure-mgmt-msi`. v0 enumerates the three audit pillars: `microsoft.compute/virtualmachines` → `azure.compute.vm`, `microsoft.storage/storageaccounts` → `azure.storage.account`, `microsoft.managedidentity/userassignedidentities` → `azure.identity.user_assigned`. Identity choice rationale: Azure AD (Microsoft Entra) principals live in Microsoft Graph API (separate SDK, separate auth domain), so Managed Identities — the workload identities, audit-relevant for compliance, addressable in Resource Graph — are the right pillar. Subscription is baked into every Resource ID (`azure-<subscription>-vm-<name>`) so multi-subscription tenants don't collide, mirroring GCP's project-in-ID and k8s's context-in-ID convention. Per-type field projection is explicit (Resource Graph returns wildly different shapes per type — `location` is a top-level column, `vm_size` is nested under `properties.hardwareProfile`, `sku` is top-level on Storage) so `graph.json` doesn't bloat with the hundreds of `properties.*` fields Resource Graph returns by default. **Resource-type allow-list validation runs BEFORE the reachability probe** — fixes a latent bug from `_build_gcp_client` where a typo'd `--resource-type` would surface as a confusing API error instead of a clean validation failure. Subscription whitespace is stripped — closes a small gap that AWS / GCP / k8s flags leave open. Auth via `DefaultAzureCredential` (env `AZURE_*` vars or `az login`); cleanly catches `ClientAuthenticationError`. Reachability probed up front by querying the first user-supplied type at `take 1` — surfaces "Microsoft.ResourceGraph provider not registered" as a clean error rather than a confusing `HttpResponseError` mid-discover. Pagination is transparent (Resource Graph caps at 1000 rows; service walks `skip_token` until exhaustion). Per-resource-type error tolerance: `403` on Compute doesn't block Storage + Managed Identity enumeration. KQL-injection note: `<resource-type>` is interpolated into the query body but is closed-allow-list-validated before reaching it; subscription never enters the KQL body (goes into the `subscriptions=[]` field on `QueryRequest`). Service at `src/lemma/services/azure_discovery.py`; CLI extension on the existing `discover_command` with new `--subscription` and `--resource-type` flags. Adds `azure-identity>=1.15` and `azure-mgmt-resourcegraph>=8.0` as core deps. Closes the entire Cloud Auto-Discovery checklist on #24 (AWS / Terraform / Kubernetes / GCP / Azure all green); only on-prem providers (vSphere / Ansible / CMDB / network scan / CSV import) plus the Core Engine items (Scope Ring Model, cross-scope evidence reuse, HCL parser, continuous validation) remain on #24.
- `lemma scope discover gcp` — fourth auto-discovery source for #24 after AWS-API (#145), Terraform-state (#146), and Kubernetes (#147). Walks Google Cloud via Cloud Asset Inventory (`cloudasset.googleapis.com`) — Google's canonical "what resources exist" API — and emits one `ResourceDefinition` per asset into the same scope-matcher → graph-write loop the AWS / TF / k8s paths use. Picks CAI over per-service APIs (compute / storage / iam_admin) because one client + one `list_assets()` call covers every GCP resource type Lemma cares about, with no per-service auth fan-out. v0 enumerates the three audit pillars: `compute.googleapis.com/Instance` → `gcp.compute.instance`, `storage.googleapis.com/Bucket` → `gcp.storage.bucket`, `iam.googleapis.com/ServiceAccount` → `gcp.iam.service_account`. Project id is baked into every Resource ID (`gcp-<project>-instance-<name>`) so multi-project orgs don't collide on the second discover run, mirroring k8s's context-in-ID convention. Service Account IDs use `--project` rather than the email's project, since Google-managed SAs (`service-12345@compute-system.iam.gserviceaccount.com`) embed a Google service project in their email — the full email is preserved under `attributes.gcp.email` for rule targeting. Attributes nest under `gcp.*` matching AWS's `aws.*` and k8s's `k8s.*`, and protobuf Structs convert via `MessageToDict(..., preserving_proto_field_name=True)` so keys are snake_case (`machine_type`, `storage_class`) matching the existing rule-writing convention. Auth via Application Default Credentials (env `GOOGLE_APPLICATION_CREDENTIALS` or `gcloud auth application-default login`); cleanly catches both `DefaultCredentialsError` and `RefreshError`. Project reachability probed up front by listing one asset of the first user-supplied type — surfaces the common "Cloud Asset API not enabled" failure as a clean error rather than a confusing per-type GoogleAPIError mid-discover. Per-asset-type error tolerance: `PermissionDenied` on Compute doesn't block Storage + IAM enumeration. Service at `src/lemma/services/gcp_discovery.py`; CLI extension on the existing `discover_command` with new `--project` and `--asset-type` flags. Adds `google-cloud-asset>=3.20` as a core dep.
- `lemma scope discover k8s` — third auto-discovery source for #24 after AWS-API (#145) and Terraform-state (#146). Walks a Kubernetes cluster's API via the official `kubernetes` Python client (added as a core dep alongside boto3) and emits one `ResourceDefinition` per discovered resource into the same scope-matcher → graph-write loop the AWS / TF paths already use. v0 enumerates three kinds — `namespace` (cluster-scoped boundary), `deployment` (declarative workload — Pods deliberately excluded because they churn within minutes and would make the graph lie), and `service` (networking entry point). Cluster context is baked into resource IDs (`k8s-<context>-deployment-<ns>-<name>`) so multi-cluster shops running prod + staging on the same kubeconfig don't corrupt each other on the second discover run. Attributes nest under `k8s.*` matching AWS's `aws.*` flat namespace, so the existing dotted-path matcher handles `k8s.labels.environment` rules unchanged. `kubectl.kubernetes.io/*` annotations are stripped before storing — `kubectl.kubernetes.io/last-applied-configuration` alone can be multi-KB and re-embeds spec data already on the resource. Auth via kubeconfig only (env `KUBECONFIG` or `~/.kube/config`); in-cluster ServiceAccount fallback is deferred behind an explicit flag in a follow-up so dev-laptop runs aren't surprised by automatic in-cluster pickup. Cluster reachability is validated up front via `version_api.get_code()` (no RBAC required) so unreachable clusters surface as clean errors. Per-kind ApiException (RBAC denial, throttling) is logged and skipped — other kinds still produce results. New flags `--context`, `--namespace`, `--kind`, plus `--dry-run`. Service at `src/lemma/services/k8s_discovery.py`; CLI extension on the existing `discover_command`. Pod / Ingress / NetworkPolicy / ConfigMap / Secret enumeration, in-cluster auth, GCP / Azure / on-prem providers, and the rest of #24's auto-discovery checklist remain on #24.
- `lemma scope discover terraform --path <terraform.tfstate>` — second auto-discovery source for #24 after the AWS-API one. Reads a Terraform state file (v4 JSON) and produces `ResourceDefinition` candidates fed into the same scope-matcher → graph-write loop the AWS path uses. Three TF types are normalized to match AWS-API discovery so existing scope rules port across sources (`aws_instance` → `aws.ec2.instance`, `aws_s3_bucket` → `aws.s3.bucket`, `aws_iam_user` → `aws.iam.user`); everything else (Google, Azure, Datadog, GitHub, etc.) keeps the raw Terraform type. Mapped types nest attributes under `aws.*` mirroring boto3's shape; unmapped types nest under `tf.*` so operators write source-specific scope rules deliberately rather than relying on silent coercion. `count` and `for_each` instance addresses get unique IDs (`tf-aws_instance.web[0]`, `tf-aws_instance.web[us-east-1a]`); `mode: "data"` resources are skipped (data sources aren't deployed assets). **Sensitive-attribute redaction** walks both `instances[].sensitive_attributes` (string + step-list path encodings) and `instances[].sensitive_values` (parallel mirror dict with `true` markers) — RDS passwords, API keys, etc. land as `"<redacted>"` rather than getting committed to `.lemma/graph.json`. Targeted not whole-block: an RDS instance with a sensitive password keeps its `engine`, `instance_class`, `tags`, so scope rules on `aws.tags.Environment` still resolve. Distinct from `lemma scope impact --plan` (plan-format JSON, gates CI on scope-boundary crossings); passing a plan file to discover exits with an error pointing at `--plan`. New service at `src/lemma/services/terraform_state.py`; CLI extension on the existing `discover_command`.
- `lemma scope discover aws` — auto-discover AWS resources into the compliance graph (Refs #24, closes the AWS-auto-discovery AC of the Living Scope Engine and the first cloud provider on the auto-discovery checklist). Walks `ec2:DescribeInstances` (paginated, with tags normalized from boto3's list-of-dicts to a plain `Key→Value` mapping so existing dotted-path scope rules like `aws.tags.Environment` work unchanged), `s3:ListBuckets` + `s3:GetBucketLocation`, and `iam:ListUsers` (paginated). For every discovered asset, runs its attributes through the existing `scope_matcher.scopes_containing()` against declared `scopes/*.yaml` and writes a `Resource` node + `SCOPED_TO` edge per match. Auth via boto3's default credential chain (env vars / profile / IMDS) with a STS `GetCallerIdentity` validation at construction time so missing credentials surface as a clean `ValueError` rather than a mid-discover stack trace. Per-service error tolerance: if one AWS API fails (AccessDenied, throttling) the others still run; the summary counts resources from successful services. `--dry-run` emits the matched `ResourceDefinition` records as YAML to stdout without touching the graph, useful as a preview before the first real run. Multi-scope matches take the alphabetically-first match and warn (single-valued `Resource → Scope` edge today; multi-scope membership is the Scope Ring Model on #24). Pruning of stale Resource nodes whose underlying asset has been deleted is tracked separately under #144. v0 ships EC2 + S3 + IAM users — RDS, Lambda, IAM roles, etc. remain on #24, as do the GCP / Azure / Kubernetes / Terraform-state / on-prem providers.
- `lemma evidence infer` — AI-propose `EVIDENCES` edges for orphaned `Evidence` nodes (Refs #88, the heuristic/LLM-mapping half). Today `lemma evidence load` only draws edges when the OCSF event carries `metadata.control_refs`; connectors that don't set the field leave Evidence nodes orphaned. `infer` walks every Evidence node with zero outgoing `EVIDENCES` edges, looks up the original event in the signed log, retrieves top-k candidate controls per indexed framework via the same vector index `lemma map` uses, and prompts the LLM for a `{confidence, rationale}` per `(evidence, candidate)` pair. Each call lands as one `AITrace` under `operation="evidence-mapping"`. Edge writes are gated by `automation.thresholds["evidence-mapping"]` (mirrors the existing `map`/`harmonize` pattern at `src/lemma/services/mapper.py:114-160`) — at or above the threshold, an `EVIDENCES` edge is written with the LLM's confidence as an edge attribute and the trace is promoted to `ACCEPTED`; below, the trace stays `PROPOSED` for human review via `lemma ai audit`. `--accept-all` overrides gating for first-time backfills. New `ComplianceGraph.add_evidence_mapping()` adds a single confidence-tagged edge without touching the Evidence node — distinct from `add_evidence()` which wipes-and-rebuilds the full edge set, so inference and load can coexist on the same Evidence without stomping each other. `EvidenceLog.get_envelope(entry_hash)` is a small reader convenience added alongside. Inference is opt-in: `lemma evidence load` stays a fast, deterministic graph sync. The AuthenticationEvent → identity-controls AC and the dedicated `evidence_query` trace discriminator AC remain tracked on #88 (the latter is itself blocked on the operation-kind discriminator in #104).
- `lemma risk load` and `lemma risk list` plus `IMPACTS` edges on resources (closes the `Risk` node and `IMPACTS` edge halves of #76 — the graph expansion epic ships its last two unchecked items). Operators author `risks/*.yaml` files (`id`, `title`, `description`, `severity`, `threatens`, `mitigated_by`) and `lemma risk load` validates each against a strict Pydantic schema (closed `RiskSeverity` enum: `low`/`medium`/`high`/`critical`) before upserting `Risk` nodes with `THREATENS` edges to Resources and `MITIGATED_BY` edges to Controls. `ComplianceGraph.add_risk()` is idempotent — re-loading rebuilds both edge sets so dropping a relationship cleanly removes the edge. Unresolved `threatens` or `mitigated_by` refs abort the batch with every miss listed and pointers at the right fix command. `lemma risk list` renders a Rich table sorted by severity (CRITICAL first), with severity color-coded (bold red / red / yellow / dim) so the worst risks read first. Adds an optional `impacts: list[str]` field to `ResourceDefinition` carrying control refs the resource directly contributes to; `lemma resource load` builds `IMPACTS` edges through `add_resource()` (semantics distinct from `SCOPED_TO` — `IMPACTS` is direct dependency, `SCOPED_TO` is scope membership). Both edge types follow the existing fail-loud-on-unresolved-refs stance.

Uploads SARIF results to GitHub Code Scanning (visible in the Security tab) and publishes the public score so the existing badge in the README resolves to a real number. Pinned action SHAs per OpenSSF guidance to avoid action-version supply-chain risk on the supply-chain-checking workflow itself.
- "Build Your First Connector" tutorial at `docs/guides/build-your-first-connector.md` (Refs #30) — end-to-end walkthrough from `lemma connector init` through `collect()` implementation, OCSF validation via `lemma connector test`, signed-evidence ingestion via `Connector.run()`, and post-hoc verification with `lemma evidence verify`. Covers the load-bearing details operators trip on (`metadata.uid` dedupe shape, `metadata.product.name` matching the manifest producer, mandatory tzinfo on `time`) and links the existing first-party connectors (GitHub, Okta, AWS) as deeper references.

- RAG evaluation harness (closes #70, a Phase 2 leftover that's been open since #17). `lemma.services.rag_eval` computes `precision@k` and `mean_reciprocal_rank` for the retrieval side of `lemma map` by running a curated corpus of `(policy_text, expected_controls)` pairs through `ControlIndexer.query_similar`. Ships with a 10-pair human-validated corpus at `tests/evaluation/corpus_nist_csf_2_0.yaml` covering every top-level NIST CSF 2.0 category. Harness unit tests (mocked indexer) run in normal CI; the real evaluation lives behind `@pytest.mark.eval` and a new `--run-eval` pytest flag so it doesn't execute without Ollama available locally. Optional `EVAL_OUT` env var appends each run's summary to a JSONL file for offline regression tracking. New `docs/guides/rag-evaluation.md` explains the metrics, baseline targets (precision@5 ≥ 0.70, MRR ≥ 0.55), and corpus-authoring guidance.
- `lemma person load` and `lemma person list` — manual Person-as-code declaration with `OWNS` edges (Refs #76, closes the `Person` node + `OWNS` edge half of the graph expansion). Operators author `people/*.yaml` files (`id`, `name`, `email`, `role`, `owns`) and `lemma person load` validates each against a strict Pydantic schema (`extra="forbid"`) before upserting `Person` nodes and `OWNS` edges into the compliance graph. The `owns` list uses the same prefixed node ids seen elsewhere in the graph — `control:<framework>:<id>` or `resource:<id>` — and a single list can mix both target types, so one edge type covers both audit semantics of "owns this control" and "owns this resource." `ComplianceGraph.add_person()` is idempotent: re-loading updates the node in place and rebuilds OWNS edges so narrowing or moving ownership cleanly drops stale bindings. Unresolved `owns` targets abort the whole batch with every missing ref named and pointers to the right fix command (`lemma framework add` for controls, `lemma resource load` for resources). `lemma person list` renders a Rich table with a ✓/✗ column showing whether every declared `owns` ref resolves in the graph, which catches drift when a resource is deleted but its owner's YAML isn't updated. Static v0 — LDAP / OIDC / HR-system auto-population stays tracked inside #76. No `init` scaffolder (real-world Person YAML is typically generated from IAM exports, a wizard is YAGNI).
- `lemma scope visualize [<scope>]` — render the scope-centered slice of the compliance graph as Graphviz DOT on stdout (closes the last named `lemma scope` CLI in #24). Includes Scope, Framework, Control, and Resource nodes with `APPLIES_TO`, `CONTAINS`, and `SCOPED_TO` edges. Pipe through any Graphviz renderer (`dot -Tpng`, `neato -Tsvg`, etc.) to turn into an image. Emitting DOT keeps Lemma free of heavy Graphviz Python bindings and lets operators pick their preferred layout. Node styling per type (scopes are rounded boxes, frameworks are 3D boxes, controls are ellipses, resources are cylinders) so the visual structure is readable at a glance. Unknown scope names exit 1 with a pointer at `lemma scope load`.
- `lemma scope posture [<scope>]` — per-framework compliance posture for declared scopes (closes the `lemma scope status displays per-framework compliance posture` AC in #24 that had been partial since #125). For each scope bound to a framework, walks `Scope → APPLIES_TO → Framework → CONTAINS → Control` and reports four counts per framework: total controls, mapped (≥1 `SATISFIES` edge), evidenced (≥1 `EVIDENCES` edge), and covered (both). No scope argument renders a summary across every scope in the graph; a specific scope argument renders a per-framework drill-down for that one scope. Read-only — emits a report, not a gate. Exits `1` on an unknown scope name (operator typo) so the command fails loud rather than silently returning empty. New `lemma.services.scope_posture` service encapsulates the traversal and returns typed `ScopePosture` / `FrameworkPosture` records, usable by future dashboard or JSON-output work.
- `lemma scope impact --plan <terraform.json>` — compute scope impact of a Terraform plan (closes the Terraform-plan half of #24's `lemma scope impact` AC). Parses Terraform plan JSON (`terraform show -json plan.tfplan`), filters out no-op rows, and for each `resource_changes[]` entry runs `change.before` and `change.after` through the matcher shipped in #132 to split the result three ways: scopes entered, scopes exited, scopes unchanged. Exits `1` when any change moves scope membership (the signal CI pipelines want — fail the build, pull a reviewer in) and `0` only when every planned change is scope-neutral. The underlying `scope_impact_for_change()` helper handles creates (no "before") and deletes (no "after") explicitly so the output is correct on those edges. New `lemma.services.terraform_plan` service contains a minimal parser that ignores everything outside `resource_changes[]`, and raises clean `ValueError`s on malformed JSON or on a file that isn't a plan (missing the `resource_changes` key — the classic "forgot `terraform show -json`" operator mistake).
- `lemma scope matches <resource-id>` — evaluate scope `match_rules` against a declared resource's attributes and print which scopes would contain it (Refs #24). Read-only; doesn't touch the graph. Useful as a sanity check before committing scope YAML or as a local preview of what a future `lemma scope impact --plan` run would trigger. The matcher service (`lemma.services.scope_matcher`) implements all four `MatchOperator` values (`equals`, `contains`, `in`, `matches`) with dotted-path traversal on `source` (`aws.tags.Environment` walks nested dicts) and a graceful empty return when the path is missing. A scope with zero rules is a catch-all that matches every resource — operators use this deliberately for org-wide scopes. This is a building block; `lemma scope impact --plan <terraform.json>` stays tracked inside #24 and will re-use the matcher when it lands.
- `lemma resource load` and `lemma resource list` — manual resource declaration (Refs #76, closes the `Resource` node + `SCOPED_TO` edge half of the graph expansion). Operators author `resources/*.yaml` files (`id`, `type`, `scope`, `attributes`) and `lemma resource load` validates each against a strict Pydantic schema (`extra="forbid"` on top-level — a typo like `resource_type:` fails loud with a line-numbered error rather than silently dropping), then adds a `Resource` node to the compliance graph with a `SCOPED_TO` edge to the named scope. `ComplianceGraph.add_resource()` is idempotent — re-adding with updated attributes updates in place and rebuilds the edge, so a resource moving to a different scope drops the old edge cleanly. Unresolved `scope:` references abort the whole batch with an error naming the missing scope, pointing operators at `lemma scope load`. `lemma resource list` renders a Rich table showing declared resources with a ✓/✗ column flagging whether each declared scope is actually in the graph. Static v0 only — auto-discovery from AWS Config, Azure Resource Graph, Terraform state, etc. stays tracked inside #24. No `init` scaffolder; real-world resource YAML is likely generated from Terraform or scripts, so a template wizard is YAGNI for v0.
- First-party AWS connector (third slice of #27; together with GitHub and Okta this closes the "at least 3 connectors fully functional" AC). `lemma.sdk.connectors.aws.AWSConnector` collects account-level posture aligned with the CIS AWS Foundations Benchmark: IAM root-account MFA enabled (CIS 1.5), IAM password policy presence + minimum length ≥ 14 (CIS 1.8), and at least one multi-region CloudTrail (CIS 3.1). Emits signed OCSF `ComplianceFinding` events via the Connector SDK. Auth via boto3's default credential chain (env vars, AWS profile, IMDS) — no Lemma-specific env var needed; missing credentials raise a clean `ValueError` at construction time via `STS:GetCallerIdentity`, not a cryptic boto3 stacktrace mid-collect. Adds `boto3>=1.34` as a core dependency. `metadata.uid` is stable per `(event_type, account_id, UTC date)` so same-day re-runs dedupe.
- `lemma evidence collect aws [--region <region>]` CLI wires the AWS connector into the existing `lemma evidence collect` entrypoint alongside `github` and `okta`.
- First-party Okta connector (second slice of #27). `lemma.sdk.connectors.okta.OktaConnector` collects identity-governance posture from an Okta org — MFA enrollment policy state and the SSO application inventory (active vs. total counts on metadata). Emits signed OCSF `ComplianceFinding` events via the Connector SDK. Auth via `LEMMA_OKTA_TOKEN` env var or constructor arg is required (Okta has no unauthenticated API); missing-token is a loud construction-time error rather than a cryptic 401 at collect time. Rate-limit responses (HTTP 429) raise a clean `ValueError` naming the endpoint. `metadata.uid` is stable per `(event_type, domain, UTC date)` so same-day re-runs dedupe against themselves.
- `lemma evidence collect okta --domain <your-org>.okta.com` CLI wires the Okta connector into the existing `lemma evidence collect` entrypoint alongside `github`.
- `lemma evidence load` — loads every envelope from the signed log into the compliance graph as an `Evidence` node, with `EVIDENCES` edges to each control named in `metadata.control_refs` (Refs #76, closes the Evidence-node half of the graph expansion; Refs #88). Operator-run, same model as `lemma scope load` and `lemma harmonize`: reads never touch the graph, mutations happen only when invoked. Idempotent — re-running updates stored node attributes in place and rebuilds `EVIDENCES` edges so a narrowed `control_refs` cleanly drops stale bindings. Fails loud on unresolved refs: a typo in a framework or control id (`nist-800-53:acc-2`) aborts the whole batch with the unresolved refs named, rather than silently dropping the edge. Introduces a `metadata.control_refs: list[str]` convention on OCSF events (shape `<framework>:<control>` per entry); the field is optional so evidence without control linkage still lands as a node, just without edges. `lemma evidence log` gains a `Graph` column (`✓` / `✗`) showing whether each envelope has been loaded.
- `lemma scope load` — wires declared scope definitions into the compliance knowledge graph (Refs #24, closes the `Scope` node half of #76). Every `scopes/*.yaml` file validated by `lemma scope status` can now be loaded as a `Scope` node with `APPLIES_TO` edges pointing at each bound framework, so `lemma graph impact scope:<name>` traverses from the scope outward through its frameworks into the controls underneath. The operation is idempotent — re-running updates each scope's `justification` and `rule_count` in place, and the edges are rebuilt so dropping a framework from the YAML drops the corresponding edge cleanly. A scope that binds to a framework not yet indexed in the graph exits `1` with an error naming the missing framework(s); no silent partial loads. `lemma scope status` gains an "In Graph" column showing `✓` / `✗` so operators can see at a glance which scopes have been loaded vs which are still YAML-only.
- Full source → normalization → storage provenance chain on every evidence envelope (closes #99). Connectors stamp a `source` record at `collect()` time naming `<producer>/<version>` as the actor; the OCSF normalizer stamps a `normalization` record via the new `normalize_with_provenance()` helper with its version and the pre-normalized payload hash; `EvidenceLog.append()` appends a final `storage` record carrying the entry hash. `lemma evidence verify` prints the chain with timestamps, actors, and truncated content hashes so an auditor can read the full story of where an entry came from without reverse-engineering the code path. The `source` and `normalization` records are included in the signed hash — tampering with either breaks verification. `lemma evidence ingest` stamps `source` with the file's SHA-256 so a flipped byte in an ingested `.jsonl` is detectable after it lands.

### Changed
- **Breaking:** `EvidenceLog`'s signed-hash algorithm now covers the pre-storage provenance records in addition to the event body. Specifically, `entry_hash = SHA-256(prev_hash || canonical_json({"event": event, "provenance": pre_storage_records}))`. Envelopes written before #99 were hashed over the event alone and will verify as VIOLATED under the new algorithm — regenerate local `.lemma/evidence/*.jsonl` files after upgrading. `EvidenceLog.append()` gains a keyword-only `provenance: list[ProvenanceRecord] | None = None` parameter; callers that don't pass it continue to get a storage-only envelope.
