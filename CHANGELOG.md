## [Unreleased]

### Added
- First-party AWS connector (third slice of #27; together with GitHub and Okta this closes the "at least 3 connectors fully functional" AC). `lemma.sdk.connectors.aws.AWSConnector` collects account-level posture aligned with the CIS AWS Foundations Benchmark: IAM root-account MFA enabled (CIS 1.5), IAM password policy presence + minimum length ≥ 14 (CIS 1.8), and at least one multi-region CloudTrail (CIS 3.1). Emits signed OCSF `ComplianceFinding` events via the Connector SDK. Auth via boto3's default credential chain (env vars, AWS profile, IMDS) — no Lemma-specific env var needed; missing credentials raise a clean `ValueError` at construction time via `STS:GetCallerIdentity`, not a cryptic boto3 stacktrace mid-collect. Adds `boto3>=1.34` as a core dependency. `metadata.uid` is stable per `(event_type, account_id, UTC date)` so same-day re-runs dedupe.
- `lemma evidence collect aws [--region <region>]` CLI wires the AWS connector into the existing `lemma evidence collect` entrypoint alongside `github` and `okta`.
- First-party Okta connector (second slice of #27). `lemma.sdk.connectors.okta.OktaConnector` collects identity-governance posture from an Okta org — MFA enrollment policy state and the SSO application inventory (active vs. total counts on metadata). Emits signed OCSF `ComplianceFinding` events via the Connector SDK. Auth via `LEMMA_OKTA_TOKEN` env var or constructor arg is required (Okta has no unauthenticated API); missing-token is a loud construction-time error rather than a cryptic 401 at collect time. Rate-limit responses (HTTP 429) raise a clean `ValueError` naming the endpoint. `metadata.uid` is stable per `(event_type, domain, UTC date)` so same-day re-runs dedupe against themselves.
- `lemma evidence collect okta --domain <your-org>.okta.com` CLI wires the Okta connector into the existing `lemma evidence collect` entrypoint alongside `github`.
- `lemma evidence load` — loads every envelope from the signed log into the compliance graph as an `Evidence` node, with `EVIDENCES` edges to each control named in `metadata.control_refs` (Refs #76, closes the Evidence-node half of the graph expansion; Refs #88). Operator-run, same model as `lemma scope load` and `lemma harmonize`: reads never touch the graph, mutations happen only when invoked. Idempotent — re-running updates stored node attributes in place and rebuilds `EVIDENCES` edges so a narrowed `control_refs` cleanly drops stale bindings. Fails loud on unresolved refs: a typo in a framework or control id (`nist-800-53:acc-2`) aborts the whole batch with the unresolved refs named, rather than silently dropping the edge. Introduces a `metadata.control_refs: list[str]` convention on OCSF events (shape `<framework>:<control>` per entry); the field is optional so evidence without control linkage still lands as a node, just without edges. `lemma evidence log` gains a `Graph` column (`✓` / `✗`) showing whether each envelope has been loaded.
- `lemma scope load` — wires declared scope definitions into the compliance knowledge graph (Refs #24, closes the `Scope` node half of #76). Every `scopes/*.yaml` file validated by `lemma scope status` can now be loaded as a `Scope` node with `APPLIES_TO` edges pointing at each bound framework, so `lemma graph impact scope:<name>` traverses from the scope outward through its frameworks into the controls underneath. The operation is idempotent — re-running updates each scope's `justification` and `rule_count` in place, and the edges are rebuilt so dropping a framework from the YAML drops the corresponding edge cleanly. A scope that binds to a framework not yet indexed in the graph exits `1` with an error naming the missing framework(s); no silent partial loads. `lemma scope status` gains an "In Graph" column showing `✓` / `✗` so operators can see at a glance which scopes have been loaded vs which are still YAML-only.
- Full source → normalization → storage provenance chain on every evidence envelope (closes #99). Connectors stamp a `source` record at `collect()` time naming `<producer>/<version>` as the actor; the OCSF normalizer stamps a `normalization` record via the new `normalize_with_provenance()` helper with its version and the pre-normalized payload hash; `EvidenceLog.append()` appends a final `storage` record carrying the entry hash. `lemma evidence verify` prints the chain with timestamps, actors, and truncated content hashes so an auditor can read the full story of where an entry came from without reverse-engineering the code path. The `source` and `normalization` records are included in the signed hash — tampering with either breaks verification. `lemma evidence ingest` stamps `source` with the file's SHA-256 so a flipped byte in an ingested `.jsonl` is detectable after it lands.
- `lemma scope init` and `lemma scope status` — scope-as-code v0 (first slice of #24). `lemma scope init [--name <name>]` scaffolds a starter `scopes/<name>.yaml` that refuses to overwrite an existing file; the starter template documents the four-field schema (`name`, `frameworks`, `justification`, `match_rules`) inline so operators can edit without leaving their editor. `lemma scope status` parses every `scopes/*.yaml`, validates against a strict Pydantic schema (`extra="forbid"` — typo'd fields fail loud rather than silently drop), and renders a Rich table of declared scopes with their framework bindings, rule counts, and audit justifications. Errors report file, line, and column via PyYAML's marked-error machinery; multiple broken files emit all their errors in one pass so operators can fix a batch in a single edit cycle. Static v0 only — auto-discovery, the scope ring model, cross-scope evidence reuse, `lemma scope impact --plan`, `lemma scope visualize`, and graph integration stay tracked inside #24.
- `lemma evidence ingest <FILE> [--dry-run]` — manual OCSF upload CLI (closes #91). Reads a `.json` single payload or `.jsonl` stream from a file or stdin (`-`), validates every record through the OCSF normalizer before any write, and appends to the signed hash-chained evidence log. Runs are all-or-nothing: a malformed record anywhere in a JSONL file rejects the whole batch with a `file:line: <error>` message so operators can fix and retry without partial writes in the log. `--dry-run` validates without appending. Dedupe on `metadata.uid` means re-running the same file is a no-op that reports the skipped count.
- `lemma check [--framework <id>] [--format text|json]` — CI/CD compliance gate (first slice of #28). Walks the knowledge graph and classifies every control in scope as `PASSED` (at least one inbound `SATISFIES` edge) or `FAILED` (none). Exits `0` only when the full selection passes; non-zero on any failure, unknown framework, or call outside a Lemma project. JSON shape (`framework`, `outcomes`, `total`, `passed`, `failed`) is frozen for downstream consumers. Text output headers the run, then tables failed controls with their short id, framework, and title. Coverage-only for v0 — SARIF (#119), the GitHub Action wrapper (#120), OPA/Rego policy-as-code (#121), and a `--min-confidence` flag (#122) ship later.
- First-party GitHub connector (first slice of #27). `lemma.sdk.connectors.github.GitHubConnector` collects repository posture from `api.github.com` — branch protection on `main`, CODEOWNERS presence, and Dependabot alert counts bucketed by severity — and emits signed OCSF evidence via the Connector SDK. Auth via `LEMMA_GITHUB_TOKEN` env var or constructor arg; unauthenticated public-repo runs work within GitHub's 60-req/hr cap. Rate-limit responses raise a clean `ValueError` naming the endpoint. Stable `metadata.uid` per `(event_type, repo, UTC date)` so re-runs the same day dedupe against themselves.
- `lemma evidence collect <connector> [--repo <owner/name>]` CLI drives first-party connectors from the shell. Ships with `github` registered; future connectors plug into the same command.
- Python Connector SDK (v0 slice of #26). Authors build connectors by subclassing `lemma.sdk.Connector`, declaring a `ConnectorManifest`, and implementing `collect()` to yield OCSF events. `Connector.run(evidence_log)` drives the stream into the append-only, signed, hash-chained evidence log using PR #100/#103 infrastructure — producer identity on the manifest is the signing identity.
- `lemma connector init <name>` scaffolds a working connector project with `manifest.json`, a `connector.py` that subclasses the reference JSONL connector, `fixtures/events.jsonl`, and a README. Refuses to overwrite an existing directory.
- `lemma connector test <path>` imports the scaffolded connector, exercises `collect()`, validates every event against the OCSF schema via the normalizer, and reports the event count. Exits non-zero on malformed input, missing file, or schema violation.
- Reference `JsonlConnector` at `lemma.sdk.reference.jsonl` — reads OCSF events from a local JSONL file. Serves as both the scaffolder's base class and a tutorial subject.
- `lemma query "<question>"` answers natural-language questions about the compliance knowledge graph. An LLM translates the question to a structured `QueryPlan` (bounded to traversals the executor supports — NEIGHBORS, IMPACT, FRAMEWORK_CONTROL_COUNT — with edge-type and direction filters), the executor walks the graph, and every invocation is recorded in the AI trace log with `operation="query"`. `--verbose` prints the resolved plan before the results so "AI that shows its work" extends to read queries too.
- Short entry-node names (e.g., `"ac-2"`) are resolved against the graph at translate time; ambiguous matches fail loud with all candidate node IDs listed. The translator retries once with validation feedback on malformed LLM output, then raises a clean `ValueError` — no silent fallback.
- New AI operation type `query` alongside the existing `map` and `harmonize` — filter with `lemma ai audit --operation query`. Query traces use `confidence=0.0` and `determination="QUERY_EXECUTED"` as conventions for the read-only operation kind.
- Key lifecycle for evidence signing. Per-producer keys now live under `.lemma/keys/<producer>/<key_id>.*.pem` with a sibling `meta.json` tracking every key the producer has held. Status is one of `ACTIVE`, `RETIRED`, or `REVOKED`. Flat pre-lifecycle layouts are auto-migrated to the versioned layout on first access.
- `lemma evidence rotate-key --producer <name>` retires the producer's active signing key and generates a fresh one. Pre-rotation entries keep verifying PROVEN under the retired key; new entries use the successor.
- `lemma evidence revoke-key --producer <name> --key-id <id> --reason <str>` marks a key REVOKED. Verification now returns VIOLATED for entries signed at or after `revoked_at` under a revoked key, and stays PROVEN for signatures made before the revocation.
- `lemma evidence keys` lists every signing key with its lifecycle state, activation timestamp, retirement/revocation timestamp, and revocation reason.
- `SignedEvidence` gains a `signed_at` field so key-lifecycle checks can compare the signing timestamp (not `event.time`) against `revoked_at`.
- Continuous proof chains on the evidence log. Every entry appended to `.lemma/evidence/YYYY-MM-DD.jsonl` is now wrapped in a `SignedEvidence` envelope with an SHA-256 hash chain (`prev_hash` → `entry_hash`) and an Ed25519 signature produced by the signing key for `metadata.product.name`. Per-producer keypairs live under `.lemma/keys/` with `0600` private-key permissions.
- `lemma evidence verify <entry_hash>` — integrity check returning PROVEN (hash + chain + signature all valid), DEGRADED (hash + chain valid but signer's key unavailable), or VIOLATED (hash or chain broken). Non-zero exit on anything other than PROVEN.
- `lemma evidence log` — Rich-table timeline showing time, class, producer, entry hash prefix, and per-row integrity state.
- `EvidenceIntegrityState` enum, `SignedEvidence` envelope model, and `ProvenanceRecord` model. The storage-stage provenance record is populated today; earlier stages populate as connectors (#26/#27) arrive.
- `lemma harmonize` now records every cross-framework equivalence decision as an `AITrace` with `operation="harmonize"`, honors `ai.automation.thresholds.harmonize` for confidence-gated auto-accept, and persists the result as an OSCAL Profile at `.lemma/harmonization.oscal.json`. The Profile imports each source catalog and encodes clusters as back-matter resources with `lemma:harmonized-cluster` properties and `rlinks` to member controls.
- `related_control_id` and `related_framework` fields on `AITrace` to carry pair-event endpoints. Map traces (single-control) default them to empty strings; harmonize traces populate both sides with a deterministic primary/secondary ordering so each equivalence is one trace.
- `lemma ai audit --operation <op>` filter for querying traces by operation type (e.g., `harmonize`, `map`).
- Auditor-facing documentation: "How Lemma's AI Works" (`docs/concepts/ai.md`) walking through framework indexing, mapping, harmonization, the PROPOSED→ACCEPTED/REJECTED lifecycle, and confidence-gated automation; a Graph Query Cookbook with 10 worked examples (`docs/guides/graph-queries.md`); a static AI System Card page (`docs/reference/ai-system-card.md`) surfacing the model registry in the published docs.
- End-to-end integration tests covering the full Phase 2 CLI pipeline: cross-artifact consistency between the compliance graph, the AI trace log, and the audit CLI; graph integrity (no orphaned policies, no phantom edges); auto-accept behavior wired through config; policy-event emission on threshold changes between `lemma map` runs.
- Shared `lemma_project` pytest fixture (`tests/conftest.py`) that lands an initialized project with `nist-csf-2.0` indexed, for use by integration tests and future PRs.
- OCSF event ingestion. `normalize()` in `src/lemma/services/ocsf_normalizer.py` dispatches an incoming payload to the right concrete OCSF model via a Pydantic discriminated-union `TypeAdapter`, rejecting payloads with a missing/unknown `class_uid` or naive (tz-less) `time`. `EvidenceLog` in `src/lemma/services/evidence_log.py` persists normalized events to `.lemma/evidence/YYYY-MM-DD.jsonl`, with `append`/`read_all`/`filter_by_class`/`filter_by_time_range` and append-time dedupe keyed on `metadata.uid` (falling back to a content hash). The log is strictly append-only — no update/delete/clear.
- OCSF (Open Cybersecurity Schema Framework) event type models at `src/lemma/models/ocsf.py`. Adds an `OcsfBaseEvent` and three concrete classes — `ComplianceFinding` (2003), `DetectionFinding` (2004), `AuthenticationEvent` (3002) — so future connectors can emit evidence in a vendor-agnostic wire format. Category consistency is enforced per class; enums use `IntEnum` to match OCSF's integer identifiers.
- `lemma ai bom` CLI command exports an AI Bill of Materials in CycloneDX 1.6 JSON format, enumerating every model in the system card with name, version, provider, purpose, training data provenance, and optional SHA hash. Output is validated against a bundled CycloneDX 1.6 structural schema before it leaves the process.
- Optional `model_hash` field on `ModelCard` for supply-chain integrity digests (e.g., `sha256:...`).
- Confidence-gated automation for AI mapping. `ai.automation.thresholds.<operation>` in `lemma.config.yaml` auto-accepts AI outputs at or above the configured threshold; outputs below remain PROPOSED for human review. Auto-accepted trace entries are marked with `auto_accepted: true` and record the applied threshold in the review rationale.
- `auto_accepted` field on `AITrace` distinguishes gate-driven acceptances from human-driven ones.
- Policy event audit log at `.lemma/policy-events/YYYY-MM-DD.jsonl`. Each `lemma map` run diffs `ai.automation.thresholds` against the prior state and appends `threshold_set`, `threshold_changed`, or `threshold_removed` events — so governance-relevant config changes are independently auditable from AI decision traces.
- Initial project scaffolding and repository structure.
- `README.md` containing the project vision, principles, and high-level architecture.
- Core governance documents including `CODE_OF_CONDUCT.md`, `CONTRIBUTING.md`, `SECURITY.md`, and `GOVERNANCE.md`.
- `ROADMAP.md` defining the Phase 0 through Phase 5 launch strategy.
- `.github` configuration containing issue templates, pull request templates, and CI/CD workflow skeletons.

### Changed
- **Breaking:** `EvidenceLog`'s signed-hash algorithm now covers the pre-storage provenance records in addition to the event body. Specifically, `entry_hash = SHA-256(prev_hash || canonical_json({"event": event, "provenance": pre_storage_records}))`. Envelopes written before #99 were hashed over the event alone and will verify as VIOLATED under the new algorithm — regenerate local `.lemma/evidence/*.jsonl` files after upgrading. `EvidenceLog.append()` gains a keyword-only `provenance: list[ProvenanceRecord] | None = None` parameter; callers that don't pass it continue to get a storage-only envelope.
