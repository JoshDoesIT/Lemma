## [Unreleased]

### Changed
- **Scope Ring Model**: a Resource can now sit in multiple overlapping scopes simultaneously (Refs #24, closes the Core Engine "Scope Ring Model" item; **unblocks** Cross-Scope Evidence Reuse — the next #24 item). Previously every Resource had exactly one `SCOPED_TO` edge and `lemma scope discover` silently degraded multi-scope matches to "first match wins" with a yellow warning. After this PR a payments DB matching both `prod-us-east` and `pci-dss` lands as **one Resource node with two `SCOPED_TO` edges**, walked by `lemma graph impact resource:<id>` to surface the union of all reachable controls. **Breaking schema change**: `resources/*.yaml` files declaring `scope: <name>` (singular) must rename to `scopes: [<name>]` (a one-element list); single-scope resources are unchanged in behavior, just renamed. The strict-`extra="forbid"` Pydantic schema produces an "extra inputs are not permitted" error naming the offending key, so operators get a one-line fix without a hand-rolled migration warning. The redundant `scope` node attribute is dropped — edges are now the only source of truth (`get_edges` is already the consumption pattern in `scope_dot`, `scope_posture`, etc.). `ComplianceGraph.add_resource` signature changes from `scope: str` to `scopes: list[str]`; idempotent rebuild semantics preserved (re-adding with `scopes=[a, c]` after `scopes=[a, b]` cleanly drops `b`). The matcher (`scope_matcher.scopes_containing`) was already returning `list[str]` since v0; the only blocker was downstream consumers, which this PR fixes. Service at `src/lemma/services/knowledge_graph.py::add_resource` + model at `src/lemma/models/resource.py`; tests in `tests/services/test_knowledge_graph.py::TestAddResource` (8 tests including N-edge creation, multi-scope rotation idempotency, all-or-nothing scope-resolution, empty-list rejection) and `tests/models/test_resource.py` (8 tests including the old-`scope:` rejection path and `min_length=1` enforcement). **Three absorbed features ship in the same PR rather than as follow-ups**: edge attribution (every `SCOPED_TO` edge carries `matched_rules: [{source, operator, value}, ...]` recording which scope rules fired during discover; manual declarations carry an empty list); the new `lemma scope explain <resource-id>` command that renders the per-scope attribution as a readable explanation (`prod ← aws.tags.Environment, equals, 'prod'`); and a new `ComplianceGraph.outgoing_edges` primitive used by `explain` and useful elsewhere. Mechanical fan-out: every existing `add_resource(scope=)` call site (11 discover services, 10+ test files) renamed to `scopes=[...]`. After this slice, the remaining Core Engine items on #24 are Cross-Scope Evidence Reuse, HCL parser, and Continuous Validation.
- `lemma query` now reaches every edge type in the compliance graph (Refs #97). The translator prompt previously capped `edge_filter` to the Phase 2 three (`SATISFIES`, `HARMONIZED_WITH`, `CONTAINS`) — a contradiction with the schema-summary block five lines below that already advertised every edge — so every relationship shipped in #76 (`EVIDENCES`, `SCOPED_TO`, `OWNS`, `IMPACTS`, `THREATENS`, `MITIGATED_BY`, `APPLIES_TO`) was invisible at the user surface. The prompt now relies on the dynamic schema summary for the vocabulary and appends a compact cheatsheet naming one example question per edge with explicit direction hints for the ones that need them (`OWNS` from a Control uses `direction=in`, etc.) plus a disambiguation note contrasting `IMPACTS` (Resource→Control) with `MITIGATED_BY` (Risk→Control). Query result rendering gains an "Attributes" column: Evidence rows show `producer · time_iso`, Risk rows show `title [SEVERITY]`, Person shows email, Resource shows its type — auditors stop hand-joining the graph export to read a single result. Filter infrastructure (time range, severity, connector source) and a dedicated `evidence_query` trace discriminator stay tracked on #97.

### Added
- **AITrace `operation_kind` discriminator + evidence-attribute query filters** (Refs #97, **closes #97 and #104**). Two coupled Phase 2 follow-ups ship together — splitting them would force #97's trace emission to carry a `# TODO: discriminator from #104` placeholder. **`AITrace.operation_kind: Literal["decision", "read"] = "decision"`** lets `lemma ai audit` separate "AI determinations that changed the compliance record" (mapper / harmonizer / evidence-mapping) from "questions someone asked" (`lemma query`) without string-matching the existing `confidence=0.0, determination="QUERY_EXECUTED"` convention. Default `"decision"` preserves every existing trace's semantics; legacy JSONL records without the field deserialize to the default cleanly so no migration script is needed. `lemma query` now stamps `operation_kind="read"` on every trace it emits, and **introduces a new `operation="evidence_query"` value** when the resolved plan carries any evidence-attribute filter — auditors can distinguish attribute-filtered evidence questions from plain graph traversals via `lemma ai audit --operation evidence_query`. **`lemma ai audit --kind decision|read`** complements the existing `--operation` flag: use `--operation` to filter to a specific kind of decision/read, use `--kind` to bisect the audit feed by intent. **`QueryPlan` gains four optional Evidence-attribute filter fields** mirroring the existing `edge_filter` any-of-list pattern: `time_range: tuple[str, str] | None` (half-open ISO-8601 `[start, end)` so adjacent queries don't double-count the boundary instant), `severity: list[str] | None` (OCSF severity *names* — "HIGH" / "CRITICAL" / ...; names beat ids for auditor readability), `producer: list[str] | None`, `class_uid: list[int] | None`. The translator prompt teaches the LLM the new vocabulary with one example per filter so natural-language questions like "high-severity GitHub auth events for AC-2 in the last 24 hours" resolve cleanly. The executor's `_neighbors_with_filters` runs a new `_passes_evidence_filters` step per node — filters apply *only* to Evidence-typed nodes; Resource / Control / Framework / Risk nodes reached by the same plan walk through unchanged because applying `time_range` to a Resource is incoherent. Both kwargs default `None` so existing call sites and persisted graphs keep working untouched. Two flags that share the same `lemma check` surface and ship together because splitting them would be merge ceremony for what's structurally one PR. **`--format sarif`** emits SARIF 2.1.0 JSON for ingestion into GitHub Code Scanning (`github/codeql-action/upload-sarif@v3`) and GitLab's SAST report path. Failures land in the Security tab alongside SAST findings with PR-aware diff annotations. Hand-rolled minimal Pydantic subset of the SARIF schema in `src/lemma/models/sarif.py` (Log/Run/Tool/Driver/Rule/Result/Location/Message — only the fields Lemma actually emits), no new external dependency. - **Continuous Scope Validation** via `lemma scope drift <provider>` and `lemma scope watch` (Refs #24, **closes the final Core Engine item — every #24 task is now green**; **Closes #144** by absorbing the stale-Resource pruning into `--apply`). Two surfaces, two distinct uses. **`lemma scope drift <provider> [provider-flags] [--apply]`** is a read-only delta detector: runs the same provider auth + discover invocation `lemma scope discover` does, then computes the per-resource SCOPED_TO delta against the current graph using the existing `scope_impact_for_change` primitive, and prints what entered scopes / exited scopes / changed attributes / was created upstream / was deleted upstream. Without `--apply` exits `1` if drift is detected (CI-friendly — pipelines can gate on drift the same way they gate on `lemma scope impact --plan`); with `--apply` mutates the graph and **prunes Resource nodes whose live infrastructure has been deleted upstream** (closes #144). Pure `compute_drift` function in `src/lemma/services/scope_drift.py` returns a `DriftReport` of `DriftEntry` records (`resource_id`, `status` ∈ `{created, deleted, scope_change, attribute_drift, unchanged}`, `entered_scopes`, `exited_scopes`, `attribute_changes` per-field before→after dict). **`lemma scope watch [--debounce-ms 300]`** is a foreground daemon (built on `watchdog`/inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) that watches `scopes/` and `resources/` for `.yaml`/`.yml`/`.hcl` changes and re-loads them on edit. Critically, when a scope file changes the daemon **re-evaluates every existing Resource against the new rules** — using stored attributes, no provider invocation needed — so YAML/HCL edits propagate to scope memberships instantly (Resources newly matching no scope are pruned; Resources newly matching different scopes get their SCOPED_TO edges rebuilt with edge attribution). Foreground-only; operators run under `tmux` / `systemctl --user` / `docker run`. `SIGINT`/`SIGTERM` shuts down the observer cleanly. **Three absorbed pieces ship together** because they share the same surface and splitting would be merge ceremony for what's structurally one feature: (1) **`lemma harmonize`-style refactor**: extract the 11-provider auth + discover chain from `discover_command` into a reusable `_build_candidates_for_provider` helper so both `discover` and `drift` consume the same plumbing without 200 lines of duplicated provider-flag logic; (2) **`ComplianceGraph.iter_resources()`**: enumerate every Resource node with reconstructed scopes (from SCOPED_TO edges) and impacts (from IMPACTS edges) — IMPACTS round-trip preserves manually-declared impacts when the watch daemon re-evaluates discovered Resources; (3) **`ComplianceGraph.remove_resource(resource_id)`**: deletion primitive that drops the node and every outgoing edge, returns True/False for existed-or-not — closes #144's pruning concern. Adds `watchdog>=3.0` as a core dep. Service primitives at `src/lemma/services/scope_drift.py` and `src/lemma/services/scope_watch.py`; CLI commands at `src/lemma/commands/scope.py::drift_command` + `watch_command` + the extracted `_build_candidates_for_provider`. Tests in `tests/services/test_scope_drift.py` (6 cycles covering created/deleted/scope_change/attribute_drift/unchanged + the `has_drift` property), `tests/services/test_knowledge_graph.py::TestIterResourcesAndRemoveResource` (5 tests including IMPACTS round-trip), `tests/commands/test_scope.py::TestScopeDrift` (5 tests covering no-drift exit 0, drift-detected exit 1, `--apply` mutation+pruning, watch helper YAML-edit re-evaluation, watch CLI signal-driven shutdown). 866 passed, 1 skipped on the full suite (was 850). **With this PR the entire #24 issue is complete**: every Auto-Discovery slice (Cloud + On-Prem), every Core Engine item (Scope-as-Code parser YAML+HCL, Scope Ring Model, Cross-Scope Evidence Reuse, Per-Scope Posture Views, HCL parser, **Continuous Scope Validation**), and every Acceptance Criterion are green.
- **HCL parser alongside YAML** for `scopes/*.hcl` and `resources/*.hcl` (Refs #24, **closes the HCL parser Core Engine item**; only **Continuous Scope Validation** remains on #24's Core Engine list). The YAML half of "Scope-as-Code parser (YAML/HCL with match rules, framework bindings, justifications)" shipped in #125; this PR closes the HCL half. - **Cross-Scope Evidence Reuse** — evidence attached to a control in one framework now implicitly satisfies its `HARMONIZED_WITH` equivalent in another framework, gated by similarity (Refs #24, **closes the Cross-Scope Evidence Reuse Core Engine item AND the matching AC** on #24). New `IMPLICITLY_EVIDENCES` edge type, distinct from `EVIDENCES`, carrying `via_control` (the harmonized peer that sourced the implicit reuse) and `similarity` (the harmonization score that gated it) for explainability. New `ComplianceGraph.rebuild_implicit_evidences(min_similarity=...)` primitive walks every Evidence node's direct `EVIDENCES` edges, follows each target control's outgoing `HARMONIZED_WITH` edges (already bidirectional via the existing `add_harmonization` shape), and writes implicit edges to peers above threshold — skipping any peer that already has a direct EVIDENCES from the same Evidence (no implicit-alongside-direct duplication; direct wins for posture math). Idempotent rebuild (drop-then-rewrite) so changing the threshold cleanly invalidates stale edges. **Four absorbed pieces ship together** because they share the same surface and splitting would be merge ceremony for flag plumbing: (1) **`lemma harmonize` now persists `HARMONIZED_WITH` edges into the compliance graph** for every pair within each cluster — the gap that surfaced during recon; the OSCAL profile + trace log were the only existing harmonization sinks, leaving the graph empty of the edges this feature needs to walk. Pair-wise edge similarity is `min(member.similarity)` (conservative — equivalent only as much as the weakest member is to the cluster head). After the edge writes, `harmonize` calls `rebuild_implicit_evidences` so reuse stays current without a separate command. (2) **`lemma evidence load` auto-rebuilds reuse** after the direct EVIDENCES walk — operators don't need to remember a separate command, and the success message reports the implicit-edge count. (3) **`lemma evidence rebuild-reuse [--min-similarity N]`** — standalone manual rebuild for operators changing the harmonization-similarity threshold out-of-band, pruning evidence by hand, or tuning reuse coverage. Reads `ai.automation.thresholds.evidence-reuse` from `lemma.config.yaml` (default 0.7 if unset) when no flag override is given. (4) **`lemma scope reuse <scope>`** — operator-facing surface that walks every implicit edge into a scope's frameworks and renders the chain (`<control> ← evidence:<hash>… (via <peer>, similarity <score>)`); distinct from `lemma scope explain <resource-id>` which explains scope-rule firing per Resource. **`lemma scope posture` extension**: the existing `Evidenced` column counts both direct + implicit (single source of truth for total coverage; what an auditor reads), with a new `Reused` column showing how many of those came via harmonization (engineering-friendly visibility into harmonization dependence; auditors typically ignore it). New `reused: int` field on `FrameworkPosture`. Threshold default 0.7 (lower than mapper's 0.85 because harmonization scores skew lower than direct mapping confidence — control families overlap, not exact paraphrase). - `lemma scope discover vsphere --host <vcenter-hostname>` — **fifth on-prem auto-discovery source for #24**, the first live SDK integration in the on-prem section after the static-file (#150) / Ansible (#151) / ServiceNow (#152) / Device42 (#153) wave. ### Changed
- **Breaking:** `EvidenceLog`'s signed-hash algorithm now covers the pre-storage provenance records in addition to the event body. Specifically, `entry_hash = SHA-256(prev_hash || canonical_json({"event": event, "provenance": pre_storage_records}))`. Envelopes written before #99 were hashed over the event alone and will verify as VIOLATED under the new algorithm — regenerate local `.lemma/evidence/*.jsonl` files after upgrading. `EvidenceLog.append()` gains a keyword-only `provenance: list[ProvenanceRecord] | None = None` parameter; callers that don't pass it continue to get a storage-only envelope.
